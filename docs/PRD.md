# PhotoMate — 产品需求文档（PRD）

> **Issue tracker**：当前仓库未配置外部工单系统。请将本文复制到你的 tracker（GitHub Issues / Linear / 飞书项目等），并打上 **`needs-triage`** 进入常规分流流程。

---

## Problem Statement

大量用户在用手机拍照时并不具备构图与曝光直觉：画面容易歪斜、主体位置不佳、曝光不合适；与此同时，手机原生相机的「专业参数」对大众用户门槛过高，缺乏「边看画面边给下一步怎么做」的实时教练体验。结果是成片率低、反复试拍成本高，也难以形成可积累的学习闭环。

---

## Solution

PhotoMate 面向「想拍出社交分享级观感但不会拍」的手机用户，提供一套 **实时取景引导 + 拍后结构化点评 + 参数建议** 的体验：

- **实时**：以低频刷新（约 2 Hz）对取景进行分析，用叠加提示（图形 + 极简文案 + 触感反馈）引导调整构图与可用控制项（曝光补偿、对焦点、变焦、HDR 等；具体以设备能力为准）。
- **拍后**：用云端多模态模型生成「定性档位 + 短总评 + 三条可执行改进」，强调正向鼓励与大白话表达。
- **端云分层**：高频、隐私敏感的取景分析优先在端侧或可控的边缘推理完成；需要语义理解与教学文案的部分走云端，并配套配额、用完即删与合规策略。
- **画质预期**：以「教练」为核心价值主张；通过 RAW / 图像增强 / 调色等手段缩小与系统相机的观感差距，但不以取代厂商计算摄影为首要目标。

## User Stories

1. 作为第一次用手机认真拍照的用户，我希望打开 App 就能看到取景画面和简单提示，以便我知道「现在该怎么挪手机」而不是瞎拍。
2. 作为经常在咖啡馆、街头拍照的用户，我希望默认不要大声语音指导，以便不打扰旁人也不尴尬。
3. 作为希望「一张就够」的用户，我希望系统在构图达标时用震动轻轻确认，以便我知道可以按快门了。
4. 作为不懂三分法术语的用户，我希望提示用大白话和数字（例如斜了几度），以便我相信建议并照做。
5. 作为容易被信息淹没的用户，我希望同一时刻只看到一条最重要的提示，以便专注调整而不是读一整屏字。
6. 作为横竖构图都会拍的用户，我希望水平线与主体位置的判定对两种方向都合理，以便我不必切换模式。
7. 作为拍人像的用户，我希望系统能识别「主体太贴边 / 头顶空间太少」等常见问题，以便我快速调整取景。
8. 作为拍风景的用户，我希望系统能提示地平线是否倾斜，以便我第一步先把画面扶正。
9. 作为弱光场景用户，我希望系统能提示曝光倾向（过亮/过暗），以便我知道该拨曝光补偿或开关 HDR。
10. 作为有 Pro 模式设备的进阶用户，我希望在支持的前提下获得 ISO、快门等组合思路，以便我在可控范围内学习参数。
11. 作为只有自动模式的用户，我希望参数建议自动降级为「能动的几项」（对焦、变焦、EV、HDR），以便我不被无效选项干扰。
12. 作为拍完就看相册的用户，我希望拍后立即得到一段简短总结和三条「下次怎么拍更好」，以便形成记忆点。
13. 作为反感打分挫败的用户，我希望用「档位」而不是冷冰冰的星级，以便我更愿意继续用下去。
14. 作为在意隐私的用户，我希望实时取景帧默认不上传云端，以便我在私人场所也敢用。
15. 作为在意隐私的用户，我希望首次启用云端点评时有明确告知与可撤回的选择，以便我对数据去向心里有数。
16. 作为在意隐私的用户，我希望照片位置信息默认被剥离后再用于云端分析，以便降低敏感元数据泄露风险。
17. 作为极简主义者，我希望提供「完全本地模式」开关，以便我可以只用端侧能力而完全不触发云端。
18. 作为数据主权敏感用户，我希望提供导出与删除与我相关的数据的入口，以便符合我对个人信息控制的预期。
19. 作为想进步的用户，我希望在 App 内看到自己历史照片的点评与时间线，以便感知「有没有在变好一点」。
20. 作为喜欢成就感的用户，我希望有轻量徽章（例如连续打卡、首次佳作），以便我愿意多打开几次 App。
21. 作为小红书用户，我希望成片观感适合社交平台传播，以便我愿意晒出来带来自然传播。
22. 作为预算敏感用户，我希望 MVP 阶段核心功能免费可用，以便我先验证是否真的有帮助再考虑付费。
23. 作为高频使用者，我希望云端点评有合理每日限额，以便产品在成本上可持续且不被滥用。
24. 作为未来付费用户，我希望订阅解锁更高限额与进阶维度分析，以便我愿意为「学得更快」付钱。
25. 作为家长/solo 创作者等垂直人群（后续），我希望一键套用场景预设（拍娃/美食/宠物等），以便少调参数多出品。
26. 作为网络不稳定用户，我希望端侧在离线时仍能给出基础构图与曝光提示，以便地铁里也能用。
27. 作为低端机用户，我希望系统在性能不足时自动降级分析频率或关闭部分维度，以便不卡顿、不过热。
28. 作为开发者/运营，我希望灰度发布与远程开关能控制新算法上线，以便出问题可快速止血。
29. 作为客服/产品，我希望崩溃率与「建议没用」反馈率可观测，以便迭代优先级有依据。
30. 作为国际化潜在用户（远期），我希望产品与合规策略可按区域切换模型与部署位置，以便扩展市场时不踩红线。

（以下为用户故事中引申出的「后台与服务」侧故事，便于研发对齐）

31. 作为客户端，我希望通过稳定的 Guidance API 获取场景分类、构图信号与参数推荐，以便 UI 只需关心展示与交互。
32. 作为编排服务，我希望能够串联「帧分析 → 规则引擎 → 文案装配 → 可选云端点评」，以便端到端延迟可控。
33. 作为风控与成本负责人，我希望匿名设备维度配额可配置，以便防止恶意刷 API。
34. 作为算法工程师，我希望构图与光线分析模块可独立替换版本，以便线上 A/B 不影响其他链路。
35. 作为内容安全负责人，我希望上传与存储链路支持短期生命周期与审计字段，以便满足合规评审。

## Implementation Decisions

- **整体形态**：Android 客户端为第一阶段交付主体；客户端内置取景与拍摄闭环；服务端提供 Guidance 能力（ unary 与 streaming 两种交互已在接口层预留）。
- **领域模块划分（逻辑视角）**：
  - **构图分析模块**：输入为主体位置与画幅信息，输出构图分项评分、总体评分与自然语言建议列表；后续可接入视觉模型输出的主体框而不改变对外契约。
  - **参数推荐模块**：输入为场景类型、光线评估与设备能力，输出 ISO、快门、光圈与白平衡等结构化建议及解释文案；需在设备能力边界内裁剪推荐区间。
  - **场景智能服务**：承担图像帧解码与并行分析（场景分类、构图视觉分析、光线评估）；通过 gRPC 与 JVM 侧服务对接。
  - **契约层**：使用 Protocol Buffers 描述 Guidance 请求/响应、设备信息、场景分析与参数推荐消息结构，便于多语言服务协同与版本演进。
  - **网关与编排**：网关对外暴露统一入口；编排服务负责聚合多源分析结果、生成主提示与次要提示列表，并记录端到端时延。
  - **用户与照片服务**：为用户标识、配额、照片元数据与点评结果持久化预留边界（具体表结构与认证方式随 MVP 后迭代）。
- **实时性**：取景侧以低频分析为原则，优先降低耗电与发热；streaming RPC 用于连续帧场景，unary 用于单帧/关键帧。
- **隐私与合规默认值**：实时帧默认本地处理；云端点评走「告知同意 → 短期存储 → 分析后删除原图」路径；剥离地理位置类敏感 EXIF 为默认策略；训练数据贡献默认关闭。
- **商业化路径**：Freemium；MVP 全免费并对云端点评设置每日软配额；后续订阅解锁更高限额与进阶能力。
- **语言与语气**：面向中文用户的大白话、命令式、可量化；避免堆砌摄影术语，教学内容优先出现在拍后点评而非实时 HUD。
- **多模块深接口**：
  - 构图分析对外保持「一次分析 → 结构化结果」的简单接口，把规则细节封装在模块内部。
  - 参数推荐对外保持「场景+光线+设备 → 参数建议」的简单接口，把档位表与边界裁剪封装在模块内部。

## Testing Decisions

- **好的测试**：只断言**对外可见行为**（评分区间、建议是否包含关键意图、推荐值是否在声明的设备范围内），不绑定私有算法细节；重构内部权重与阈值时测试仍应稳定。
- **优先测试的模块**：
  - 构图分析引擎（三分法评分、对称度、边缘警告、无主体提示等）。
  - 参数推荐引擎（不同场景与光线下的 ISO/快门等裁剪与解释文案）。
  - Guidance 契约的兼容性与字段必填语义（通过契约测试或生成代码的编译约束体现）。
  - 配额与网关路由的行为（集成测试阶段接入）。
- **测试风格**：JUnit 5 + Truth（已在仓库采用）；新测试沿用「行为描述型」用例命名。
- **刻意不测或弱化单测的部分**：原生相机驱动的端到帧流水线、第三方视觉模型的准确率（改用离线评测集与线上监控）；UI 像素级快照测试不作为 MVP 硬性门槛。
- **先行验收路径**：为核心引擎维持小而快的单元测试套件，使之在 CI 中可频繁运行；对 Python 场景服务补充面向纯函数的组件测试与最小化集成测试（随服务硬化逐步补齐）。

## Out of Scope

- **iOS 客户端**：第一阶段不做；可在验证 Android PMF 后启动。
- **完全替代系统相册管理**：不做重型相册产品；仅保留与「教练闭环」相关的最小历史记录能力。
- **社交社区**：不做内置社交网络；传播依赖系统分享渠道。
- **复杂人像贴纸/美颜流水线**：不作为 MVP 核心；若有需求以可选扩展评估。
- **任意机型全参数可控**：不承诺所有设备可调 ISO/快门/光圈；以设备能力探测为准自动降级。
- **跨境默认使用海外多模态模型**：国内市场默认采用合规区域内的模型与部署；海外版本单独评估。

## Further Notes

- 当前仓库已具备 **Kotlin 构图与参数规则引擎雏形**、**Guidance 相关 protobuf 契约**、以及 **Python 场景智能服务的骨架**（含并行分析编排思路）；Android App 壳层与完整云端链路仍需按里程碑补齐。
- PRD 中的「实时叠加交互」「触感反馈」「档位化拍后点评」「配额与订阅」等条款需在客户端与网关层逐项落地验收。
- 若组织开展 grill-me / eng review，可将「双轨练习/出片」类诉求统一收敛为「教练闭环优先 + 画质增强开关」，避免产品叙事分裂。
- 发布到 issue tracker 时：建议拆分为「客户端 MVP」「scene-intelligence 硬化」「网关配额与合规」「数据留存与徽章」等 epic，再进入开发排期。
