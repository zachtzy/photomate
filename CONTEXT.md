# PhotoMate

A real-time photography coaching app that guides users through viewfinder overlays before the shot and structured critiques after it.

## Language

**Guidance**:
The composite result returned to the client for a single frame: composition score, parameter suggestions, and prioritized tips.
_Avoid_: advice, recommendation (as a top-level concept)

**Composition Analysis**:
Evaluation of how well a frame's subject placement follows classical photography rules (rule of thirds, symmetry, leading lines).
_Avoid_: composition scoring, framing analysis

**Parameter Suggestion**:
A set of camera setting recommendations (ISO, shutter speed, aperture, white balance) tailored to scene, lighting, and device capabilities.
_Avoid_: parameter recommendation (reserved for the protobuf message name)

**Scene Type**:
A classification of the photographic subject or environment (e.g. landscape, portrait, night, street).
_Avoid_: scene category, mode

**Lighting Condition**:
A structured assessment of the ambient light: direction, quality (hard/soft), color temperature, dynamic range, and an overall score.
_Avoid_: light analysis, exposure assessment

**Device Capabilities**:
The hardware limits reported by the client device (ISO range, aperture range, OIS, sensor size, focal length) used to clamp parameter suggestions.
_Avoid_: device profile, camera specs

**Critique**:
A post-capture evaluation produced by a cloud multimodal model: a rating tier, a short summary, and three actionable improvement suggestions.
_Avoid_: review, feedback, score report

**Rating Tier**:
A qualitative level (not a numeric star rating) assigned to a captured photo in a **Critique**, designed to encourage rather than discourage. Three tiers: 「可以更好」(room to grow), 「不错」(solid), 「赞」(great). The lowest tier is constructive, never punishing.
_Avoid_: star rating, score, grade

**Tip**:
A single actionable instruction displayed to the user during viewfinding (e.g. "主体过于靠近边缘"). Only one **Primary Tip** is shown at a time; others are **Secondary Tips**.
_Avoid_: suggestion (overloaded), hint, advice

**Tip Prioritizer**:
An on-device component that ranks all tips from composition, parameter, and lighting analyses and selects the single **Primary Tip** to display.
_Avoid_: tip selector, suggestion ranker

**Quota**:
A daily limit on cloud **Critique** requests, tracked per anonymous device installation ID. No user account required for MVP.
_Avoid_: rate limit (reserved for abuse prevention at the gateway level)

**Tilt**:
The rotational deviation of the device from level, measured in degrees via the device's accelerometer/gyroscope. A high-priority real-time tip signal.
_Avoid_: horizon, level (as standalone terms)

**Ready Signal**:
A haptic confirmation fired when the composition score exceeds a configurable threshold AND tilt is within tolerance. Tells the user "you can press the shutter now."
_Avoid_: shutter confirmation, good-enough signal

## Relationships

- A **Guidance** result aggregates one **Composition Analysis**, one **Lighting Condition**, one **Parameter Suggestion**, and one **Scene Type**
- A **Parameter Suggestion** is clamped by **Device Capabilities**
- **Composition Analysis** runs on-device (Kotlin `core-analysis`) for real-time viewfinder guidance at ~2 Hz
- Real-time **Guidance** is fully on-device for MVP; no server calls during live viewfinding
- On-device **Guidance** includes both **Composition Analysis** and **Parameter Suggestion** (with `UNKNOWN` scene type, providing lighting-driven generic recommendations)
- **Scene Type** classification is server-side only for MVP; on-device uses `UNKNOWN` as default
- On-device **Lighting Condition** is derived from EXIF heuristics (ISO, EV, white balance); full ML-based lighting assessment is server-side only
- On-device subject detection uses ML Kit Object Detection (single-object mode) to produce **SubjectPosition** for **Composition Analysis**
- **Tilt** is detected via independent SensorManager listener (~10-30 Hz), stored in AtomicReference; **Tip Prioritizer** reads latest tilt on each frame analysis cycle; tilt also drives UI level indicator independently
- **Ready Signal** fires when composition score exceeds threshold AND **Tilt** is within tolerance
- **Critique** is a separate cloud API, distinct from the real-time **Guidance** pipeline, with its own latency and privacy constraints
- A **Critique** is generated synchronously: client uploads photo → Photo Service strips geolocation and stores with short TTL → **Guidance Orchestrator** calls Scene Intelligence + multimodal LLM → Critique returned in same request → photo deleted after TTL
- The **Guidance Orchestrator** owns both future server-side guidance and **Critique** generation; **Photo Service** handles photo storage/retrieval only
- **Critique** is generated by Qwen-VL (Alibaba DashScope API) as the domestic China-compliant multimodal model for MVP
- **Critique** pipeline is sequential: Scene Intelligence produces structured analysis → Orchestrator sends photo + structured context to Qwen-VL → LLM generates natural-language **Critique** grounded in the analysis
- The **Guidance Orchestrator** is server-side only; it calls **Scene Intelligence** via gRPC and does NOT depend on `core-analysis`
- The **Tip Prioritizer** runs on-device and selects a single **Primary Tip** from all analysis results
- Server-side services (Gateway, Guidance Orchestrator, User Service, Photo Service) are deployed as independent microservices; Python **Scene Intelligence** is a separate service communicating via gRPC

## Flagged ambiguities

- "CompositionAnalyzer" exists in both Kotlin (`core-analysis`) and Python (`scene-intelligence`) — resolved: Kotlin is the on-device real-time engine; Python is the server-side ML-powered engine for post-capture.
- `guidance-orchestrator` depended on `core-analysis` — resolved: this dependency should be removed; the orchestrator is server-side only and calls scene-intelligence via gRPC.
- Identity for MVP is anonymous device ID only; user accounts, history timelines, and badges are deferred to post-MVP.
- Proto messages (server wire format) and Kotlin domain models (on-device) are intentionally different contracts; they do not need to align.
- When multiple subjects are detected on-device, the largest/most-confident detection is used as the single main subject for **Composition Analysis**.
- `param-recommendation` module is dead code (empty shell); `ParameterAdvisor` lives in `core-analysis`. Module should be deleted.
- "Local mode" for MVP simply hides the **Critique** upload; real-time **Guidance** is already fully on-device.
- Cloud **Critique** consent dialog appears on first shutter tap; remembered per device.
- Photo storage uses Alibaba Cloud OSS (via S3-compatible API/AWS SDK) for domestic compliance, aligned with Alibaba Cloud stack (Qwen-VL/DashScope).
- Shared PostgreSQL instance with separate schemas per service for MVP; user-service and photo-service each own their schema.

### On-device frame analysis pipeline

- CameraX `ImageAnalysis` with `STRATEGY_KEEP_ONLY_LATEST` + single-thread custom Executor
- Pipeline per frame: ML Kit Object Detection → `CompositionAnalyzer` → EXIF-based `LightingCondition` → `ParameterAdvisor` → `TipPrioritizer`
- Hard 500ms budget; if exceeded, result is discarded and UI keeps previous **Guidance**
- Results emitted via Kotlin `StateFlow<GuidanceResult>`
- UI: Jetpack Compose + Canvas overlays (rule-of-thirds grid, subject box, level indicator, tip text) with CameraX `PreviewView` via `AndroidView` interop
- Android `app` module added to monorepo, depends on `core-analysis` (pure Kotlin/JVM, no Android SDK dependency); all Android-specific code (CameraX, ML Kit, SensorManager, Compose) in `app`

## Example dialogue

> **Dev:** "用户打开取景器，**Guidance** 是怎么产生的？"
> **Domain expert:** "CameraX 每帧送到单线程 Executor，ML Kit 先检测主体，然后 **CompositionAnalyzer** 算构图分数，EXIF 推出 **Lighting Condition**，**ParameterAdvisor** 用 `UNKNOWN` 场景给出通用参数建议，最后 **Tip Prioritizer** 选出一条 **Primary Tip** 显示。整条链 500ms 硬上限。"
>
> **Dev:** "用户按快门之后呢？"
> **Domain expert:** "如果用户已同意云端点评，照片上传到 **Photo Service**，剥离地理位置后存到 OSS。**Guidance Orchestrator** 先调 **Scene Intelligence** 拿结构化分析，再把分析结果 + 照片发给 Qwen-VL 生成 **Critique**——一个 **Rating Tier**、一段总评、三条改进建议——同步返回给客户端。照片在 TTL 后删除。"
>
> **Dev:** "如果用户不想用云端呢？"
> **Domain expert:** "实时 **Guidance** 本身就全在端上，不需要网络。关掉云端只是隐藏 **Critique** 入口。"
